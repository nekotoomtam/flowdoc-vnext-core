# Artifact Job Execution Slice

Status: Phase 177 artifact job execution slice.

This phase adds an internal-alpha executor in `@flowdoc/internal-alpha-runner`
that turns an artifact job request into minimal PDF spike bytes, filesystem byte
storage, a rendered artifact manifest, and a rendered artifact job record.

It is an external package boundary. It does not make `@flowdoc/vnext-core`
depend on concrete storage, the PDF spike package, filesystem writes, workers,
queues, backend routes, or production renderer behavior.

## Scope

- Create a queued artifact job and planned artifact manifest from the public
  core artifact job contract.
- Persist the planned manifest and queued job through `@flowdoc/storage-file-json`.
- Advance the job through layout and rendering lifecycle transitions.
- Execute the existing minimal text-only PDF spike package.
- Write produced PDF bytes through the filesystem artifact byte store.
- Build a rendered artifact manifest from the byte-store digest and storage key.
- Complete and persist the rendered artifact job.
- Read records and bytes back enough to produce JSON-safe execution evidence.
- Persist a failed artifact job and failed manifest when the PDF spike blocks.

## PASS

- Phase 177 runs outside `@flowdoc/vnext-core` in
  `packages/internal-alpha-runner`.
- The executor consumes public core contracts plus the external storage and PDF
  spike packages.
- Successful execution writes and reloads:
  - planned, rendering, and rendered artifact manifest records;
  - queued and rendered artifact job records;
  - PDF bytes in the filesystem byte store.
- Rendered manifests are checked against stored bytes for artifact id,
  byteLength, sha256, and storageKey consistency.
- Failed PDF spike execution is captured as a bounded failed job and failed
  manifest, not as a production retry queue.
- The report is JSON-safe and does not include raw PDF bytes.

## FAIL-BLOCKER

None for the narrow internal-alpha slice.

Any blocked artifact job creation, storage conflict, byte-store failure, or PDF
spike blocker is surfaced as bounded execution evidence.

## RISK

- Record writes and artifact byte writes are not transactionally linked.
- The PDF bytes are generated by the minimal text-only spike, not a production
  fidelity renderer.
- The executor is a direct helper, not a queue worker or backend route.
- Retry scheduling is represented by failed job records only.

## UNKNOWN

- Production PDF renderer package choice.
- Worker/queue ownership and retry backoff.
- Backend route execution and auth/authz boundary.
- Multi-record atomicity strategy.
- Whether internal alpha should continue hardening the minimal spike or move to
  a production renderer package.
